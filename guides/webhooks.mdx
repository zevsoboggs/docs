## Главное

<Warning>
**Вебхуки настраиваются только через [веб-кабинет](https://loveandpay.io/dashboard/webhooks)**.
Публичного API для создания/изменения/удаления вебхуков **нет** — это сделано осознанно для предотвращения атак на ваш endpoint через утёкший API-ключ.
</Warning>

В этом руководстве — как создать вебхук в UI, какие события доступны, как проверять HMAC-подпись и обрабатывать retry-логику.

## Создание вебхука

<Steps>
  <Step title="Откройте раздел в кабинете">
    [Панель управления → Вебхуки](https://loveandpay.io/dashboard/webhooks)
  </Step>

  <Step title="Нажмите «Создать вебхук»">
    Заполните форму:

    - **URL** — куда отправлять POST. Должен быть **HTTPS** в production.
    - **События** — отметьте нужные галочки (см. список ниже).
    - **Описание** (опционально) — для своих заметок.
  </Step>

  <Step title="Скопируйте секрет">
    После создания система покажет **Webhook Secret** — уникальный ключ для этого вебхука. Сохраните его — он нужен для проверки HMAC-подписи входящих запросов.

    <Note>Секрет можно посмотреть позже в карточке вебхука — он **не одноразовый**, в отличие от API-секрета.</Note>
  </Step>

  <Step title="Проверьте доставку">
    В карточке вебхука есть кнопка **«Тест»** — отправит проверочный POST на ваш URL с фиктивным payload'ом. Если у вас 2xx — вебхук работает.
  </Step>
</Steps>

## Требования к URL вебхука

<Warning>
URL вебхука должен вести на **ваш собственный домен**. Анонимные сервисы-приёмники
запрещены — их используют для мошеннических схем, поэтому такие URL **отклоняются**.
</Warning>

Требования к адресу:

- **Только HTTPS** — `http://` не принимается.
- **Домен, а не IP** — голый IP-адрес указать нельзя, только доменное имя.
- **Не приватные/внутренние адреса** — `localhost`, `127.0.0.1`, приватные подсети и т.п. заблокированы.
- **Запрещены анонимные приёмники и тоннели**: `webhook.site`, RequestBin, Beeceptor, Hookbin,
  Pipedream, `ngrok`, `localtunnel`, `trycloudflare` и подобные.
- **Привязка к подтверждённому домену.** Если у вашего аккаунта включена
  [верификация доступа к API](/guides/api-access-verification), хост вебхука обязан совпадать
  с вашим **подтверждённым доменом** (или быть его поддоменом) — например, `https://api.ваш-домен.com/webhooks`.

<Info>
Если URL не проходит проверку, API/панель вернёт ошибку `400` с понятной причиной
(например, «Анонимные сервисы-приёмники запрещены. Укажите ваш собственный домен»).
Укажите эндпоинт на своём домене — и всё заработает.
</Info>

## Доступные события

**Счета:**

| Событие | Когда срабатывает |
|---|---|
| `invoice.created` | Счёт создан |
| `invoice.updated` | Счёт обновлён |
| `invoice.paid` | Клиент оплатил счёт |
| `invoice.expired` | Истёк час, счёт стал EXPIRED |
| `invoice.cancelled` | Счёт отменён |

**Транзакции:**

| Событие | Когда срабатывает |
|---|---|
| `transaction.created` | Создана платёжная транзакция |
| `transaction.completed` | Транзакция завершена (оплата прошла) |
| `transaction.failed` | Транзакция отклонена |

**Плательщик (анти-фрод):**

| Событие | Когда срабатывает |
|---|---|
| `payment.payer_info` | После оплаты получены данные плательщика (ФИО/телефон от провайдера) |
| `payment.third_party_payer` | Счёт оплачен **третьим лицом** (ФИО плательщика ≠ ожидаемого) |

**KYC:**

| Событие | Когда срабатывает |
|---|---|
| `kyc.started` | Клиент начал верификацию |
| `kyc.approved` | KYC успешно пройден |
| `kyc.rejected` | KYC отклонён |
| `kyc.expired` | Срок KYC-сессии истёк |

**Возвраты:**

| Событие | Когда срабатывает |
|---|---|
| `refund.requested` | Создана заявка на возврат (через API или кабинет) |
| `refund.completed` | Администратор утвердил, возврат исполнен ✅ |
| `refund.failed` | Возврат не удалось исполнить (см. `executionError`) |
| `refund.rejected` | Администратор отклонил заявку |

<Note>
Подробнее о самом процессе — в разделе [Возвраты](/guides/refunds).
</Note>

## Формат payload

POST на ваш URL с JSON-телом и заголовками:

```http
POST https://your-server.com/lovepay-webhook HTTP/1.1
Content-Type: application/json
X-Webhook-Event: invoice.paid
X-Webhook-Signature: 7d2a8b3c...
X-Webhook-Timestamp: 2026-05-20T14:00:00.000Z
X-Webhook-Id: 4f8e9a1d-xxxx-...

{
  "event": "invoice.paid",
  "timestamp": 1716207600000,
  "invoice": {
    "id": "uuid",
    "invoiceNumber": "INV-260520-0042",
    "amount": 150050,
    "currency": "RUB",
    "status": "PAID",
    "paidAt": "2026-05-20T14:00:00.000Z",
    "metadata": { "orderId": "12345" },
    "customerEmail": "user@example.com",
    "customerPhone": "+79001234567"
  }
}
```

| Заголовок | Описание |
|---|---|
| `X-Webhook-Event` | Тип события (`invoice.paid` и т.д.) |
| `X-Webhook-Signature` | HMAC-SHA256 подпись **тела запроса** (hex) |
| `X-Webhook-Timestamp` | Время отправки, ISO 8601 |
| `X-Webhook-Id` | ID вебхука (для логов/диагностики) |

<Note>
`amount` — в копейках (RUB × 100). Поле `metadata` всегда возвращается из того, что вы передали при создании счёта.
</Note>

## Проверка HMAC-подписи (обязательно!)

Перед обработкой **всегда** проверяйте `X-Webhook-Signature` — иначе любой может вам слать фейковые "оплачено".

<Warning>
Подпись считается от **сырого тела запроса** (без timestamp):
`signature = HMAC-SHA256(webhook_secret, raw_body)` в hex.
Проверяйте именно **необработанные байты** тела (до парсинга JSON) — повторная
сериализация изменит строку и подпись не сойдётся.
</Warning>

<CodeGroup>

```javascript Node.js
const crypto = require('crypto')

app.post('/lovepay-webhook', express.raw({ type: 'application/json' }), (req, res) => {
  const signature = req.headers['x-webhook-signature']
  const body = req.body.toString('utf8') // сырое тело

  const expected = crypto
    .createHmac('sha256', process.env.WEBHOOK_SECRET)
    .update(body)
    .digest('hex')

  if (
    signature.length !== expected.length ||
    !crypto.timingSafeEqual(Buffer.from(signature), Buffer.from(expected))
  ) {
    return res.status(401).end('Invalid signature')
  }

  const event = JSON.parse(body)
  console.log('Получили:', event.event, event.invoice?.invoiceNumber)

  // ⚠️ Отвечайте 200 OK сразу — обрабатывайте асинхронно
  res.status(200).end('ok')
  queue.add('process-lovepay', event)
})
```

```python Python (Flask)
import hmac
import hashlib
import os
from flask import Flask, request, abort

app = Flask(__name__)
SECRET = os.environ['WEBHOOK_SECRET']

@app.post('/lovepay-webhook')
def webhook():
    signature = request.headers.get('X-Webhook-Signature', '')
    raw_body = request.get_data(as_text=True)  # сырое тело

    expected = hmac.new(
        SECRET.encode(),
        raw_body.encode(),
        hashlib.sha256
    ).hexdigest()

    if not hmac.compare_digest(signature, expected):
        abort(401, 'Invalid signature')

    event = request.get_json()
    # ⚠️ Сразу 200, обработка асинхронно
    queue.enqueue(process_event, event)
    return 'ok', 200
```

```php PHP
<?php
$secret = getenv('WEBHOOK_SECRET');
$signature = $_SERVER['HTTP_X_WEBHOOK_SIGNATURE'] ?? '';
$body = file_get_contents('php://input'); // сырое тело

$expected = hash_hmac('sha256', $body, $secret);

if (!hash_equals($expected, $signature)) {
    http_response_code(401);
    exit('Invalid signature');
}

$event = json_decode($body, true);
http_response_code(200);
echo 'ok';
// Дальше — отправляйте на воркер для обработки
```

</CodeGroup>

## Доставка и retry

Наш бэк отправляет вебхук **сразу** при событии. Таймаут — **10 секунд**. Если получили не 2xx или request упал, делаем повторы по экспоненте:

| Попытка | Через |
|---|---|
| 1 | сразу |
| 2 | +1 минуту |
| 3 | +5 минут |
| 4 | +15 минут |
| 5 | +1 час |
| 6 | +4 часа |

После 6-й неуспешной попытки вебхук помечается как `failed`, но можно вручную перепослать через **«История доставки»** в карточке вебхука.

<Tip>
**Идемпотентность**. При ретраях одно и то же событие может прийти повторно. Дедуплицируйте по паре `invoice.id` + `event` (или по `invoiceNumber` + `status`) — сохраняйте обработанные и игнорируйте дубли.
</Tip>

## Best practices

- **Отвечайте 200 OK немедленно**. Обработку (письма клиенту, обновление БД) делайте асинхронно в очереди. Если ответите долго — у вас таймаут, у нас ретрай.
- **Только HTTPS** в production. HTTP вебхуки технически работают, но Cloudflare/Mintlify-провайдеры могут блокировать.
- **Логируйте `X-Webhook-Id` и `invoiceNumber`** во всех своих логах — пригодится при разборе спорных случаев с поддержкой.
- **Проверяйте signature ВСЕГДА**, не верьте даже если IP белый.
- **Замена секрета**: в кабинете есть «Сгенерировать новый секрет» — старый перестаёт работать сразу, обновите его на своей стороне без даунтайма.

## История и отладка

В кабинете в [Вебхуки → история доставки](https://loveandpay.io/dashboard/webhooks) видно:

- Время отправки и попыток
- HTTP-статус ответа
- Тело ответа (первые 1 КБ)
- Возможность **повторить вручную**

## FAQ

<AccordionGroup>
  <Accordion title="Можно ли создавать вебхуки через API?">
    Нет. Только через веб-кабинет. Это политика безопасности — чтобы скомпрометированный API-ключ не позволил атакующему перенаправить ваши финансовые уведомления.
  </Accordion>
  <Accordion title="Куда придёт webhook если URL временно недоступен?">
    Будем повторять до 6 раз с возрастающим интервалом (см. таблицу выше). Все попытки видны в истории доставки.
  </Accordion>
  <Accordion title="Можно ли иметь несколько вебхуков?">
    Да. Можно создать N вебхуков на разные URL и разные наборы событий — например, один для production, второй для staging.
  </Accordion>
  <Accordion title="Что отвечать если события не нужно обрабатывать?">
    Любой 2xx ответ считается успешным. Если конкретное событие не интересно — просто верните 200 без обработки.
  </Accordion>
  <Accordion title="Есть тестовый webhook для разработки?">
    Используйте кнопку **«Тест»** в карточке вебхука — отправит примерный payload на ваш URL. Или подключите [webhook.site](https://webhook.site) для отладки.
  </Accordion>
</AccordionGroup>
